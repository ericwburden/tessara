# Tessara Naming Decisions & Implications

---

## 1. Final Project Name

### ✅ Selected Name
**Tessara**

---

## 2. Rationale

### 2.1 Evocative, Not Descriptive
“Tessara” is derived from *tessera* (a piece of a mosaic), but is intentionally modified to:

- increase uniqueness
- improve brand ownership
- avoid overly literal or academic tone
- enable stronger domain availability

---

### 2.2 Conceptual Alignment with Architecture

The name maps cleanly to the system’s core ideas:

| Concept | System Feature |
|--------|------|
| Mosaic / composition | Dashboards, reports |
| Individual tiles (tesserae) | Nodes, submissions, data points |
| Structured assembly | Hierarchy + reporting engine |
| Pattern formation | Aggregation + analytics |

---

### 2.3 Phonetic Strength

Compared to alternatives (e.g., Looma):

- easier to say
- more natural cadence
- stronger product-name feel
- less ambiguity in pronunciation

Example usage:

- “We’re building this in Tessara”
- “That lives in Tessara”
- “Open it in Tessara”

---

### 2.4 Brandability

“Tessara” is:

- unique enough to define internally
- flexible across domains (not tied to non-profits only)
- suitable for long-term productization
- not overly technical or overly playful

---

## 4. Product Naming System

### 4.1 Functional Module Names

- **Tessara Core** → platform foundation
- **Tessara Nodes** → hierarchy system
- **Tessara Forms** → form builder
- **Tessara Flow** → submissions + workflows
- **Tessara Insight** → reporting engine
- **Tessara Atlas** → dashboards

---

### 4.2 Alternative (More Abstract System)

- **Tessara Core**
- **Tessara Weave**
- **Tessara Pattern**
- **Tessara Lens**

---

### 4.3 Naming Guidelines

- Use **Tessara** as the root prefix for all modules
- Avoid overly technical suffixes (e.g., “Service”, “Engine”)
- Favor conceptual names over literal ones
- Maintain consistency across UI and codebase

- reporting
