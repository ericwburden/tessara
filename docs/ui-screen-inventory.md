## Migration guardrails

- Active routes start as native Leptos components returning `impl IntoView`.
- No route migrates string-rendered UI, transitional shell code, or broad feature files.
- `#app-overlays` is a direct child of `body` before `#app-root`.
- Tailwind 4 is the styling target; Bulma assets and shell CSS have been removed from the active reset baseline.
- Lists default to shared table primitives unless a route has a specific UX reason to diverge.

Sweep note: on 2026-05-17, the DTO cleanup removed workflow-level form anchors, renamed workflow scope to `workflow_node_type_id`, moved workflow revision authoring to explicit ordered steps, and removed persisted form section column counts. Workflow-mediated assignment is now the assignment model: assignable work is represented by `workflow_assignments`, while "Assign Form" is a UI convenience path that creates or reuses a generated single-step workflow and then uses normal workflow assignment mechanics.

Workflow history note: during the reset pass, workflow history is product-facing "revision" language even though API paths, DTO types, and database tables still use `version` names for compatibility. Step changes create a new integer workflow revision label; workflow name, node type, and description changes update metadata without creating a new revision.

Workflow scope note: workflows are now available at an explicit list of concrete nodes, and workflow assignments are created only from those available workflow/node combinations. Form versions are not intrinsically tied to node types for workflow compatibility. Workflow steps should become the owner of target/context semantics: a step may target the assigned workflow node, a node selected or derived from prior step data, no node context, or a future external/context object. Future step metadata and context passing should support nonlinear branching and forward data mapping instead of using form-level or workflow-level node type scope as the compatibility rule.

Workflow assignment note: `form_assignments` and `submissions.assignment_id` have been removed. Submissions carry their concrete response target through `workflow_assignment_id`, `workflow_instance_id`, `workflow_step_instance_id`, `form_version_id`, and `node_id`. Workflow records now include `source` and `source_form_id`; `source = 'generated_form'` marks visible single-form shortcut workflows, and editing such a workflow into anything other than the exact single-form shortcut promotes it to `authored` so the next Assign Form shortcut creates a fresh generated workflow.

Response access note: response reads, draft saves, submits, deletes, and starts are assignment-backed. Submission access joins through `submissions.workflow_assignment_id`; respondent access is granted to the workflow assignment assignee or to an account delegating for that assignee. Admin and scoped-operator access remain capability/scope gates, but no response access path depends on direct form assignment storage.

Response start options note: `/api/responses/options` is intentionally assignment-only. It returns pending workflow assignment start choices for the current account or an accessible delegated account. It must not grow a form/node manual-start mode again; form-first starts should go through the generated single-form workflow shortcut and then start a workflow assignment.
